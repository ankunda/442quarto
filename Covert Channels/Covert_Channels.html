A Covert Channel in cybersecurity is a method used to secretly transfer information in a way that violates security policies. These channels operate by exploiting legitimate communication or computing mechanisms in unintended ways, allowing unauthorized data exfiltration without detection.

Covert channels typically imply the existence of an overt channel — a legitimate, sanctioned communication channel that is being exploited to carry the hidden message. Examples of overt channels include websites, FTP servers, phone conversations, chat applications, social media feeds, and TCP/IP packets.

⚡ Challenge Yourself

Can you think of ways that the above overt channels could be used to communicate covertly? Consider code words, preset conversation systems, puzzles, number ciphers, or steganographic embedding. The more mundane the channel, the harder it is to detect the hidden message.

Because typical covert channels allow communication between entities that are supposed to be able to talk to each other — as permitted by some access control policy — they tend to be:

  • Hidden
  • Hard to detect
  • Require admin access to set up
  • Low bandwidth
  • Resource-intensive

That last point — low bandwidth — is important. A covert channel typically requires a great deal of time or computation relative to the amount of information it can secretly transmit. Sending a single ASCII character may require many overt messages or significant timing overhead.

Steganography
Hides data within existing data (e.g. an image). Requires a pre-existing communication channel.
vs.
Covert Channel
Creates a channel using unintended side effects of a system's normal operation. No pre-existing channel needed.
📌 Note

Steganography (hiding data within data) is not the same as a covert channel. The key distinction is that steganography requires a prior communication channel — you must already be able to send a file or message — while covert channels create a channel from scratch using the side effects of system behavior. Both are clandestine; they achieve it differently.

Covert channels fall into two main categories, each with a distinct mechanism:

Storage Channels

Communication via modification of a shared storage resource. Sender writes; receiver reads.

Timing Channels

Communication via measurable changes in system timing or resource availability.

1 Storage Covert Channels

In a storage covert channel, the sender (or sending process) modifies a shared resource that everyone has legitimate access to. The receiver then reads from that same resource to extract the hidden message. No direct communication between sender and receiver is needed.

Here are three concrete examples of how a storage covert channel might be constructed:

  • 1

    Abusing the print queue. Everyone with access to a system can observe the printer job queue. A sender process can submit jobs to the queue to signal a 1, or deliberately skip submissions to signal a 0. A receiver polling the queue at regular intervals can reconstruct the bitstream and decode the message over time.

  • 2

    Abusing web server log files. Web servers record every page request. A sender could access a designated page to transmit a 0, and a different page to transmit a 1. A receiver with read access to the log file can then parse the access pattern and reconstruct the hidden message — with no suspicious direct communication between parties.

  • 3

    FTP file permissions. FTP servers are easy to deploy and loosely supervised. A sender can embed a message in the permission bits of files on the server. Casual users see only a file listing; a receiver who knows the encoding scheme can read the hidden message from the permission bits. Each file's 10-character permission string can carry up to 10 bits of information.

    drwxrwxrwx 1 prof prof 0 Mar 26 17:52 'Some random file name'

    With the simple scheme "letter present = 1, dash = 0," each file contributes up to 10 bits to the covert message.

2 Timing Covert Channels

Timing covert channels rely on measurable variations in system behavior over time rather than any observable data stored somewhere. The receiver must watch the system and measure intervals — it never reads a file or accesses a log directly.

  • 1

    File access duration. A sender opens and closes a file for varying durations: a short hold signals 0, a long hold signals 1. The receiver, watching the file lock or access time, accumulates bits and decodes the message. This can work entirely within a single machine.

  • 2

    Port knocking patterns. Port knocking (covered later in the course) is a technique for keeping network ports hidden until a secret knock sequence is recognized. However, a sender can deliberately use an incorrect knock pattern to encode bits. A receiver inside the network observes the pattern and decodes the message — even though the ports remain closed and traffic appears normal from the outside.

  • 3

    Hard drive acoustic side channel. Mechanical hard drives produce distinct sounds depending on head movement direction and speed. Researchers demonstrated that these sounds can be deliberately controlled by a process to transmit data from an air-gapped computer — one completely disconnected from the internet. A nearby device with a microphone records and decodes the audio, exfiltrating data without any network connection.1

1 Guri et al., "DiskFiltration: Data Exfiltration from Speakerless Air-Gapped Computers via Covert Hard Drive Noise." arXiv:1608.03431. arxiv.org/abs/1608.03431
🔬 Lab Connection

The Timing Covert Channel Lab will have you build a TCP server that transmits an overt text message character-by-character, with the delay between each character encoding a hidden binary message. Short delay = 0; long delay = 1.

3 Network Covert Channels

Network covert channels are not a separate fundamental category — any example here could be classified as either storage or timing — but they share a common feature: exploitation of network infrastructure as the carrier.

A classic example is embedding a covert message in the header fields of network packets. The overt payload (the body of the packet) carries legitimate-looking data; security systems typically inspect the payload for suspicious content. However, certain header fields — IP identification numbers, TCP sequence numbers, TTL values, reserved bits — receive far less scrutiny. A sender places bits of the covert message in these fields; a receiver who knows where to look can reconstruct the hidden message, packet by packet, as the traffic flows through the network.

Because normal traffic is the cover story, these channels are extremely difficult to distinguish from legitimate network activity without deep packet inspection policies specifically designed to detect covert encoding.
