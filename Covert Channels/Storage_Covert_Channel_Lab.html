In the lecture, you covered different ways that someone might set up and exploit a covert channel. In this lab, you will build one from scratch — a storage covert channel that hides a secret message inside the file permission bits of an FTP server.

To an ordinary observer browsing the FTP server, everything looks normal. But someone who knows the encoding scheme can read the hidden message simply by examining which permission bits are set on each file.

1 Setting Up an FTP Server

💡 Tip

Depending on how far you got in a previous lab, you may not need to repeat the FTP server setup. Skip ahead to Section 2 if your server is already configured and running.

1.1 Installation

You will use vsftpd (Very Secure FTP Daemon), a lightweight and widely-used FTP server for Linux. Install it with the following commands:

sudo apt update          # refresh the package index
sudo apt install vsftpd  # install vsftpd

Once installed, verify that vsftpd is running and listening on the network:

sudo netstat -alpn | grep ' LISTEN '
💡 Tip — netstat not found?

If netstat isn't available, install it with:
sudo apt install net-tools
Then re-run the netstat | grep command above. Look for a line containing vsftpd to confirm it's active.

1.2 Logging In

Try connecting to your own FTP server from the same machine:

ftp localhost

This will prompt for a username and password — use your system credentials. Once connected, explore your available files using standard FTP commands:

ls         # list files
ls -lh     # list files with permissions and sizes
cd folder  # change into a directory
cd ..      # go up one level
Ctrl + D   # disconnect

1.3 Configuration

Make a few tweaks to improve the experience. Open the vsftpd configuration file:

sudo vim /etc/vsftpd.conf   # or use nano, gedit, etc.

Find and adjust the following settings:

anonymous_enable=YES
local_enable=NO
ftpd_banner=My First FTP Server   # customise as you like
anon_root=/home/ftp               # create this directory if it doesn't exist

Then restart the server to apply your changes:

sudo service vsftpd restart
ftp localhost

This time you should be able to log in anonymously — username anonymous, blank password (just press Enter). You can also connect from another machine on the same network using the server's IP address:

ftp 138.47.x.x   # replace with your server's IP address
💡 Virtual Machine Tip

If your FTP server is running in a VM with a bridged or NAT network adapter, the VM will have its own IP address distinct from the host. You can FTP into it from the host machine using that IP.

2 Storage Covert Channel

Now for the interesting part. Linux file permissions consist of 10 characters — a file type indicator followed by three groups of read/write/execute flags for the owner, group, and others:

Permission String Anatomy
example
d rw- r-- r--
meaning
type owner group other
as bits
? 110 100 100 → 9 usable bits (+ type flag)

The convention is simple: any letter present = 1; a dash = 0. This gives us up to 10 bits of information per file. The two labs below show two different ways to exploit this.

2.1 The 7-Bit Scheme

Standard ASCII characters use 7 bits. We can encode one character per file by using the last 7 permission bits (the group and other permission bits), and deliberately leaving the first 3 bits (the owner permissions) unset. The file type character is ignored.

Files carrying part of the message will have no owner permissions set. To avoid suspicion, we intersperse noise files — files where at least one of the first 3 bits is set. Noise files are skipped by the receiver.

Let's walk through encoding the word "Demigod":

CharDecimal7-bit9-bit (with 00 prefix)chmod
D68100010000 100 010 0104
e101110010100 110 010 1145
m109110110100 110 110 1155
i105110100100 110 100 1151
g103110011100 110 011 1147
o111110111100 110 111 1157
d100110010000 110 010 0144

The 9-bit representation groups neatly into three octal digits (owner / group / other), which is exactly how chmod expects its argument. Create the files and set permissions:

touch file1 && chmod 104 file1
touch file2 && chmod 145 file2
touch file3 && chmod 155 file3
touch file4 && chmod 151 file4
touch file5 && chmod 147 file5
touch file6 && chmod 157 file6
touch file7 && chmod 144 file7

The resulting directory listing looks like this — note that no message file has any owner permission set:

---x---r-- 1 prof prof 0 Mar 27 16:15 file1
---xr--r-x 1 prof prof 0 Mar 27 16:15 file2
---xr-xr-x 1 prof prof 0 Mar 27 16:15 file3
---xr-x--x 1 prof prof 0 Mar 27 16:15 file4
---xr--rwx 1 prof prof 0 Mar 27 16:15 file5
---xr-xrwx 1 prof prof 0 Mar 27 16:15 file6
---xr--r-- 1 prof prof 0 Mar 27 16:15 file7

Now add noise files (each with at least one owner permission bit set) to make the listing look natural:

---x---r-- 1 prof prof 0 Mar 27 16:15 file1
d--xrw-r-- 1 prof prof 0 Mar 27 16:15 file1.5    ← noise
---xr--r-x 1 prof prof 0 Mar 27 16:15 file2
---xr-xr-x 1 prof prof 0 Mar 27 16:15 file3
-r-xrwxrwx 1 prof prof 0 Mar 27 16:15 file3.5    ← noise
---xr-x--x 1 prof prof 0 Mar 27 16:15 file4
---xr--rwx 1 prof prof 0 Mar 27 16:15 file5
-rwx--xr-x 1 prof prof 0 Mar 27 16:15 file5.5    ← noise
---xr-xrwx 1 prof prof 0 Mar 27 16:15 file6
-rw-r---wx 1 prof prof 0 Mar 27 16:15 file6.5    ← noise
---xr--r-- 1 prof prof 0 Mar 27 16:15 file7
drwxr-xrw- 1 prof prof 0 Mar 27 16:15 file7.5    ← noise

To decode, a receiver scans each file, skips any with an owner permission bit set (noise), and reads the remaining 9 bits as a character:

---x---r-- → 0001000100 → bits 1–9 = 001000100 → D
d--xrw-r-- → first char is 'd' (owner bit set) → SKIP (noise)
---xr--r-x → 0001100101 → e
...and so on

2.2 The 10-Bit Scheme

The 7-bit scheme wastes the first 3 permission bits. With a small change, we can use all 10 bits — squeezing more data out of each file.

The trade-off: without those "empty owner bits" to distinguish message files from noise files, we can no longer include noise. Every file in the listing carries part of the message, and the receiver processes them all.

  • 1

    Sort alphabetically. The receiver must process files in a consistent order. Alphabetical filename order provides a natural, reproducible sequence.

  • 2

    Flatten the bitstream. Convert the entire covert message to binary, concatenate all bits into one long string, then split into groups of 10. Assign one group per file.

  • 3

    Pad if necessary. If the message doesn't divide evenly into groups of 10, pad the end with zeros.

  • 4

    Decode in 7-bit chunks. The receiver collects all bits in order, splits them into 7-bit groups (standard ASCII), and converts each group to its character.
